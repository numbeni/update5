# Product Check Monitoring Integration Upgrade

Before making any changes, thoroughly inspect the entire codebase and understand how the current monitoring system, diagnostics system, live console, logging, incidents, alerts, dashboard, site details, sweep engine, manual checks, scheduled checks, and Product Check implementation currently work.

Do not assume anything.

Review the existing architecture first and then design the best possible implementation.

Use your judgment to integrate this feature in the cleanest, most maintainable, most scalable, and most performant way possible.

---

## Existing Product Check Logic Must Remain Unchanged

The current Product Check implementation already works correctly and is already used inside Diagnostics.

This existing implementation must remain the single source of truth.

Do not redesign it.

Do not replace it.

Do not simplify it.

Do not create a second Product Check implementation.

Do not create a lightweight version.

Do not create a monitoring-specific version.

Do not duplicate any Product Check logic.

When Product Check is executed during monitoring, it must use the exact same Product Check workflow that is currently executed inside Diagnostics.

This includes:

* Product discovery logic
* Homepage discovery
* Sitemap discovery
* Product URL extraction
* Product URL selection
* Product validation
* Product success/failure determination
* Existing response validation rules
* Existing diagnostics behavior

The monitoring system should invoke the existing Product Check process rather than implementing a new one.

The goal is very simple:

Run the existing Diagnostics Product Check automatically during monitoring whenever Product Check is enabled for a site.

There must be only one Product Check implementation in the entire project.

Diagnostics and Monitoring must always use the same underlying Product Check logic so results remain consistent.

# Main Goal

Today Product Check is mostly used through Diagnostics.

We want Product Check to become part of the monitoring lifecycle for sites where it is enabled.

Whenever a site has Product Check enabled:

* Scheduled monitoring runs must execute Product Check.
* Sweep monitoring runs must execute Product Check.
* Manual monitoring runs must execute Product Check.
* Retry/Recheck monitoring runs must execute Product Check.
* Any monitoring workflow that evaluates a site's health should also evaluate Product Check.

The Product Check should continue using the existing discovery and validation logic.

Only one valid product sample is required per monitoring cycle.

The goal is simply:

"Can at least one real product page load correctly?"

Product Check activity must be visible in the Live Console in real time.

Not after completion.

Not only in diagnostics.

When a site is being monitored and Product Check is enabled, the operator should be able to see Product Check activity occurring live.

# Monitoring Behavior

For sites where Product Check is enabled:

1. Run the normal monitoring process.
2. Calculate the site's normal monitoring status.
3. Execute the existing Product Check logic.
4. Validate one discovered product URL using the existing Product Check implementation.
5. Use both results to derive the final status.

Product Check should run automatically whenever monitoring runs.

This includes:

* Scheduled monitoring
* Sweep monitoring
* Manual checks
* Retry checks
* Rechecks
* Any workflow that evaluates a site's health

---

# Status Behavior

Homepage Status + Product Status should produce meaningful final statuses.

If homepage is healthy and product page is healthy:

→ Existing behavior remains unchanged.

If homepage is healthy but product page fails:

→ Create and surface a Product Page Issue state.

Users should immediately understand:

"The website is reachable, but product pages are failing."

If homepage fails but product page succeeds:

→ Create and surface a Homepage Issue / Product OK state.

Users should immediately understand:

"The homepage is failing, but product pages still work."

If both fail:

→ Existing DOWN behavior remains.

You may improve naming if a better naming convention exists in the project.

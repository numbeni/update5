# 1. Fix monitoring status logic

## Problem

Currently, the system treats `degraded` as a failure, which causes sites to be marked as `down` even when they are actually reachable.

Example:

* HTTP status is 200
* DNS is OK
* SSL expires soon
* Check result becomes `degraded`
* But `degraded` increases `consecutiveFailures`
* Eventually the overall site status becomes `down`

This is incorrect.

## Expected behavior

Clearly separate **reachability** from **quality issues**.

### Reachable states — NOT failures

These should NOT increase `consecutiveFailures`:

```text
up
slow
degraded
```

These mean the site is accessible.

### Failure states

Only real unreachable states should increase `consecutiveFailures`, such as:

```text
down
timeout
dns_failure
connection_failed
tcp_unreachable
fetch_failed
```

## Required changes

Find logic like:

```ts
if (raw !== "down" && raw !== "degraded") return raw;
```

Change it to:

```ts
if (raw !== "down") return raw;
```

Find logic like:

```ts
if (status === "down" || status === "degraded") {
  consecutiveFailures++;
}
```

Change it to:

```ts
if (status === "down") {
  consecutiveFailures++;
} else {
  consecutiveFailures = 0;
}
```

Whenever a check result is:

```text
up
slow
degraded
```

reset:

```ts
consecutiveFailures = 0;
```

Final rules:

```text
down      = truly unreachable
degraded  = reachable but has issues, such as SSL expiring soon
slow      = reachable but high latency
up        = fully healthy
```

After this fix:

* SSL warnings should result in `degraded`, not `down`
* A site that responds with HTTP 200 must not be marked as `down`
* `down` should only represent real outages

Apply this across all relevant monitoring logic, database updates, site status derivation, incident logic, and API responses.

# 2. Curl-style URL diagnostic check

Add a diagnostic feature similar to a simplified `curl`.

## Purpose

Quickly show whether a URL is reachable and what HTTP response it returns.

## Logic

For each monitored site or manually requested URL:

* Send an HTTP request
* Follow redirects
* Capture:

  * original URL
  * final URL after redirects
  * HTTP status code
  * response time in ms
  * redirect count
  * content-type
  * server header if available
  * status group: `2xx`, `3xx`, `4xx`, `5xx`
  * ok boolean
  * error message if request fails

Example output:

```json
{
  "url": "https://example.com",
  "finalUrl": "https://www.example.com",
  "statusCode": 200,
  "responseTimeMs": 140,
  "redirectCount": 1,
  "contentType": "text/html",
  "server": "nginx",
  "statusGroup": "2xx",
  "ok": true,
  "errorMessage": null
}
```

Important:

* Do not mark the whole site as down only because this diagnostic fails.
* This is a diagnostic tool, not the main status source.
* Add a manual UI button: `Run Curl Check`.

# 3. Product / Category availability check

Add an optional ecommerce diagnostic check based on my old `unixcheck.py` logic.

## Purpose

A shop homepage may be reachable, but product/category pages may be broken. This check detects that.

## Logic

For a site:

1. Fetch homepage HTML.
2. Extract links from `<a href="">`.
3. Look for ecommerce-related patterns:

```text
/product
/products
/item
/shop
/store
/category
/categories
/cat
/collection
/collections
/cart
/checkout
```

4. Normalize relative links into absolute URLs.
5. Randomly sample 3 to 5 discovered URLs.
6. Test each sampled URL with a lightweight HTTP request.
7. Consider the check successful if at least one sampled product/category URL returns:

```text
200
301
302
```

8. If no product/category links are found on the homepage, try common sitemap URLs:

```text
/product-sitemap.xml
/sitemap_products_1.xml
/sitemap.xml
```

9. Extract URLs from sitemap XML and test product/category-like links from there.

## Output

```json
{
  "enabled": true,
  "status": "ok",
  "productPagesFound": true,
  "source": "homepage",
  "checkedUrls": [
    "https://shop.com/product/example-item",
    "https://shop.com/category/mobile"
  ],
  "workingUrls": [
    "https://shop.com/product/example-item"
  ],
  "message": "Product/category pages are accessible"
}
```

## Status meanings

```text
ok       = at least one product/category URL works
warning  = product/category links were found but some failed
failed   = no product/category pages could be reached
unknown  = check disabled or not enough data
```

Important:

* This check must be optional per site.
* Not every monitored site is an ecommerce site.
* Add field such as:

```ts
productCheckEnabled: boolean
```

* If disabled, do not run it automatically.
* Do not mark the whole site as down because product/category pages fail.
* Show it as diagnostic/warning only.

# 5. Better bulk import initial checks

Currently, when many sites are bulk imported, the app appears to auto-check them too aggressively and many sites get marked down.

## Required behavior

When bulk importing many sites:

* Do not fire all initial checks at once.
* Queue them.
* Check them one by one or with very small concurrency.
* Add delay between checks.

Suggested behavior:

```text
concurrency: 1 or 2
delay between checks: 2–5 seconds
```

This should reduce false negatives caused by DNS/network overload.

Important:

* Bulk import should still return a summary immediately.
* Initial checks can run in the background.
* Log progress clearly.
* Do not crash if one check fails.

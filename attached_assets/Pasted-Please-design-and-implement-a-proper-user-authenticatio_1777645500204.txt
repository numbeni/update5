Please design and implement a proper user authentication and user panel system for this existing Hosting Manager / NOC Monitoring project.

Important:
- Do not rewrite the whole app.
- Keep the current architecture.
- Add authentication and user management cleanly into the existing backend/frontend/database structure.
- Use secure and production-friendly practices.
- Do not hardcode passwords.
- Do not expose secrets to frontend.

Project context:
- Hosting / NOC Monitoring Dashboard
- Backend: Express + TypeScript
- Frontend: React + Vite
- Database: PostgreSQL + Drizzle
- Existing app has dashboard, monitored sites, incidents, queue, logs, settings, charts, and operators will use it.

Goal:
When a user opens the website for the first time, they should be required to log in with username/email and password before accessing the NOC dashboard.

Add a complete but simple user system.

Required features:

1. Login system

Add a login page.

Fields:
- Email or username
- Password

Behavior:
- If user is not logged in, redirect to login page.
- After successful login, redirect to dashboard.
- If login fails, show a clear error message.
- Add loading state while logging in.
- Password field should be hidden by default with optional show/hide button.

Security:
- Passwords must be hashed in database.
- Never store plain text passwords.
- Use secure password hashing such as bcrypt or argon2.
- Use session cookie or JWT safely.
- Prefer httpOnly cookie if practical.
- Add logout support.

2. First admin user setup

Since this project may not have users yet, add a first-time setup flow.

Behavior:
- If no users exist in database:
  - show “Create Admin Account” page
  - ask for:
    - first name
    - last name
    - email
    - username
    - password
    - confirm password
  - create the first admin user
- After first admin is created, disable public registration unless enabled by admin.

This prevents being locked out after deployment.

- last name

Password change:
- current password
- new password

6. Admin user management panel

Add a Users / Operators management page for admins.

Admin can:
- View users
- Create new user
- Edit user
- Disable/enable user
- Change user role
- Reset password or set temporary password

Do not allow deleting the last admin.

Roles behavior:
Admin:
- Full access
- Manage users
- Manage settings
- Delete sites
- Clear history
- Run checks
- Pause/resume monitoring

Operator:
- Use dashboard
- Run checks
- Manage incidents
- View logs
- Add to queue
- Generate reports
- Cannot manage users
- Cannot change global dangerous settings unless allowed

Viewer:
- Read-only access
- Can view dashboard, charts, incidents, logs
- Cannot run checks
- Cannot delete
- Cannot pause monitoring
- Cannot change settings

7. Permission system

Add backend permission checks.

Important:
- Do not rely only on frontend hiding buttons.
- Backend APIs must enforce permissions too.

Examples:
- Only admin can manage users.
- Only admin/operator can run checks.
- Only admin/operator can resolve incidents.
- Only admin can delete monitored sites or clear history.
- Viewer cannot perform write actions.

Frontend:
- Hide or disable buttons based on role.
- Show clear message if user has no permission.

- Make logout invalidate the session/token.
- Never log passwords.

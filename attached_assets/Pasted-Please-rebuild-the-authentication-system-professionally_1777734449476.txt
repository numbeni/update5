Please rebuild the authentication system professionally and cleanly from the beginning for this existing Hosting Manager / NOC Monitoring project.

Current problem:
The current auth/login system is not reliable. For example, after setting a password, login sometimes says the password is wrong. Some auth flows are not logical.

Goal:
Create a clean, secure, predictable authentication system with first-time founder registration, then normal login for all users.

Important:
- Do not rewrite the whole app.
- Keep the current project architecture.
- Replace/fix the current auth implementation cleanly.
- Remove broken auth logic if needed.
- Use secure password hashing.
- Do not store plaintext passwords.
- Do not expose secrets to frontend.
- Keep TypeScript clean.

Project context:
- Backend: Express + TypeScript
- Frontend: React + Vite
- Database: PostgreSQL + Drizzle
- App type: Hosting / NOC Monitoring Dashboard
- Existing roles may include admin/operator/viewer.

1. First-time founder registration

When the website is opened and there are no users in the database:

- Show a first-time setup page.
- This page should create the first user as the Founder/Admin.
- This setup should happen only when user count = 0.

Founder setup fields:
- First name
- Last name
- Display name
- Email
- Username
- Password
- Confirm password

Validation:
- All fields required.
- Email must be valid.
- Username must be unique.
- Password minimum 8 characters.
- Password and confirm password must match.

After successful setup:
- Create the first user with role:
  founder or admin
- Prefer role: founder if role system supports it.
- If not, use admin with isFounder = true.
- Automatically log in the founder.
- Redirect to dashboard.

Important:
This setup page must NOT appear again after the first user is created.

2. Normal login after setup

After the founder account exists:

- Opening the app should show the login page if user is not authenticated.
- No public registration page should be available.
- Only founder/admin can create other users from inside the panel.

Login fields:
- Username or email
- Password

Behavior:
- If credentials are valid → login and redirect to dashboard.
- If invalid → show a clear error.
- Do not say whether username/email or password is wrong separately.
- Use generic error:
  “Invalid username/email or password.”

3. Fix password hashing and verification

This is critical.

Implement password handling correctly and consistently.

Requirements:
- Use bcrypt or argon2.
- On user creation:
  - hash password once
  - store only passwordHash
- On login:
  - find user by username OR email
  - verify submitted password against passwordHash
- Do not double-hash passwords.
- Do not compare plaintext password with hash.
- Do not accidentally store the password field instead of passwordHash.
- Make sure setup, user creation, reset password, and change password all use the same hash/verify functions.

Add tests or debug-safe checks if possible.

4. Session/auth token system

Implement a reliable session system.

Preferred:
- httpOnly cookie session

Requirements:
- Cookie name: noc_token or similar
- httpOnly: true
- sameSite: lax
- secure: true in production, false in local dev if needed
- maxAge: 7 days or configurable
- path: "/"

Backend:
- Login sets session cookie.
- Logout clears session cookie.
- Auth middleware reads cookie.
- Auth middleware also optionally supports Bearer token for API testing.

Frontend:
- All auth fetch requests must use:
  credentials: "include"

Important:
Make sure CORS/cookie settings work correctly in Replit and local development.

5. Auth endpoints

Add/fix these endpoints:

GET /api/auth/setup-status
Returns:
{
  "setupRequired": true | false
}

POST /api/auth/setup
Creates first founder/admin user only if no users exist.

POST /api/auth/login
Logs in existing user.

POST /api/auth/logout
Logs out current user.

GET /api/auth/me
Returns current authenticated user.

POST /api/auth/change-password
Authenticated user changes their own password.

Optional admin endpoints:
GET /api/users
POST /api/users
PATCH /api/users/:id
PATCH /api/users/:id/role
PATCH /api/users/:id/status
POST /api/users/:id/reset-password

6. Route protection

Frontend:
- Before showing app routes, call /api/auth/setup-status.
- If setupRequired = true → show Founder Setup page.
- Else call /api/auth/me.
- If authenticated → show app.
- If not authenticated → show Login page.

Protected routes:
- Dashboard
- Charts & Status
- Monitored Sites
- Incidents
- Logs
- Live Console
- Settings
- DNS Performance
- User management
- Profile
- Site detail pages

Unauthenticated users should only see:
- Founder setup page if no users exist
- Login page if setup already completed

7. User roles and permissions

Implement or fix roles cleanly.

Roles:
- founder
- admin
- operator
- viewer

If founder role is too much, use:
- role = admin
- isFounder = true

Recommended permissions:

Founder:
- Full access
- Cannot be deleted
- Can manage admins/operators/viewers
- Can access all settings

Admin:
- Manage users except founder restrictions
- Manage settings
- Delete sites
- Clear history
- Run checks
- Pause/resume monitoring
- Manage DNS resolvers

Operator:
- Use dashboard
- Run checks
- Manage incidents
- Add to queue
- Generate reports
- View logs
- View DNS Performance
- Cannot manage users
- Cannot delete important settings unless allowed

Viewer:
- Read-only
- Can view dashboard/charts/sites/incidents/logs
- Cannot run checks
- Cannot delete
- Cannot pause/resume
- Cannot manage users/settings

Important:
- Enforce permissions in backend.
- Frontend hiding buttons is not enough.

Add/fix Users management page.

Founder/admin can:
- Create operator/viewer/admin accounts

- Do not allow non-founder admins to remove founder privileges.

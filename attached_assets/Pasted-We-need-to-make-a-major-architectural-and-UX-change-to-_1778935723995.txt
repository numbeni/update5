We need to make a major architectural and UX change to the NOC monitoring platform.

This platform is used by a hosting/server company. We host ecommerce websites on multiple internal hosting servers. From now on, the monitoring system must become server-based, sweep-based, and operator-first.

The current flat site monitoring flow is no longer enough. Operators need to understand website health by hosting server, not just by individual websites.

This redesign is not only a UI change. It changes the way monitoring is executed, displayed, prioritized, and trusted.

The system must be designed for real Iranian network conditions, where temporary routing issues, packet loss, ISP instability, and short-lived failures can create false alarms. The goal is to improve both accuracy and operator usability.

Focus on this new server-based monitoring architecture, grid-first dashboard, live monitoring state, and execution model.

Do not rewrite the entire project from scratch. Refactor the existing architecture intelligently and preserve working features where possible.

==================================================
MAIN GOAL

Transform the monitoring system from:

Flat Website Monitoring

into:

Server-Based Hosting NOC Monitoring

Every monitored website must belong to exactly one hosting server.

The dashboard, monitoring execution logic, status bar, live progress indicators, and site status calculations must all be redesigned around servers.

The main/default operator experience must be Grid View.

Grid View is the primary view of the system.

Compact View and List View must use the exact same underlying site status data and filtering logic as Grid View.

If a site is red/down in Grid View, it must also be down in Compact and List View.

If a site is recovered, slow, degraded, unstable, or up in Grid View, every other view must reflect the same status.

There must be one single source of truth for site status.

Do not create separate status logic for Grid, List, and Compact modes.

==================================================

2. Remove Queue From Main Monitoring Flow

Remove the current queue-based automatic monitoring flow from the main dashboard and monitoring process.

The old queue should no longer control automatic monitoring.

The new automatic monitoring flow must be:

Server-Based Sweep Monitoring

There should be no visible Monitoring Queue section in the main dashboard for this flow.

The system should support:

automatic full sweep
manual full sweep
manual single-site check
manual single-server check if useful

But the main automatic monitoring process must no longer be queue-based.

The execution should be deterministic and easy for operators to understand.

Operators must know:

which server is being checked now
which site is being checked now
which phase is running
when the next sweep will happen
whether monitoring is paused
whether manual checks are running

8. New Monitoring Sweep Logic

Replace the old automatic monitoring process with a server-based sweep.

A full sweep means:

go server by server
check all sites inside one server
finish that server
then move to the next server

Do not mix checks between sites or between servers.

Very important rule:

For each site, run all checks for that site first, then move to the next site.

Correct:

Server s1:
Site A:
DNS
HTTP
TCP
Curl / availability check
Site B:
DNS
HTTP
TCP
Curl / availability check

Wrong:

DNS for all sites
HTTP for all sites
TCP for all sites

The old behavior caused confusion with many sites and must be avoided.

Tests must not overlap in a way that causes unclear or mixed status updates.

This is very important for operator trust.

==================================================
9. Accuracy For Iranian Network Conditions

The monitoring engine must be optimized for unstable Iranian network conditions.

Do not mark a site as critical down after only one failed check.

Temporary failures should not immediately become critical alerts.

Use confirmation logic:

first server pass
second pass for down sites inside the same server
final global down recheck

Only persistent failures should remain in Critical / Down Sites.

This reduces false alarms caused by temporary routing problems, DNS instability, ISP issues, and short connection failures.

The system must balance:

accuracy
speed
low false positives
clear operator feedback
predictable execution
==================================================
10. Server First Pass

For each server:

mark that server as currently checking
show live indicator on that server header
show phase:

First try on s1

Then check all sites belonging to that server, one by one.

After all sites of that server are checked:

update site statuses
update server metrics
identify down sites in that server

Server metrics must update based on the same status source used by Grid/List/Compact.

==================================================
11. Server Second Pass For Down Sites

After the first pass of one server finishes:

If any sites are down in that server, run a second check only for those down sites.

Show phase:

Second try on s1 down sites

If a site is still down after the second pass:

move it to the top Critical / Down Sites banner
keep its server reference
show a small server badge on the critical card

If the site recovers:

keep it in its server section
update status normally

This second pass is important because Iranian network conditions can cause temporary false failures.

==================================================
12. Cooldown Between Servers

After finishing a server, wait a short cooldown before moving to the next server.

Default cooldown:

1 or 2 seconds

This value must be configurable in Settings.

During cooldown, show:

Cooldown before next server: 2s

Then continue to the next server.

This cooldown should also be used as a safe moment for non-critical background tasks where appropriate.

==================================================
13. Final Global Down Recheck

After all servers have completed their first and second passes:

Run one final recheck on all sites currently shown in the Critical / Down Sites banner.

Show phase:

Last checking on down sites

If a critical site is now Up or Slow:

remove it from Critical banner
return it to its original server section
update server metrics

If it is still Down:

keep it in Critical banner

After this final check, the full sweep is complete.

==================================================
14. Sweep Completion Behavior

When a sweep starts, show:

When a sweep completes, show:

Last sweep completed X minutes ago
Next sweep in X minutes

The next sweep interval must come from Settings.

Add a button:

Run Full Sweep Now

This button should start a full sweep immediately, even if the next scheduled sweep is not due yet.

If a sweep is already running, prevent duplicate execution and show:

Sweep already running

==================================================
15. Monitoring Pause Behavior

When monitoring is paused:

no automatic sweep runs
no background checks run
no automatic retries run
no automatic recovery checks run
no scheduled site checks run

Only manual actions should work:

run check on one site
run check on one server
run full sweep manually if explicitly allowed
diagnostics requested by the operator

The UI must clearly show that monitoring is paused.

Very important:

If monitoring is paused, the system must not continue checking sites in the background.

If all sites are deleted, automatic monitoring must fully stop.

If there are no monitored sites, no background monitoring activity should continue.

If all servers are empty, no sweep should run.

If the site list becomes empty while a sweep is running, stop the sweep safely and clear live execution state.

==================================================
16. Manual Checks Must Have Higher Priority

Manual checks triggered by an operator must have higher priority than automatic monitoring.

Examples:

manual check from DNS section
manual check from server section
manual check from any troubleshooting panel

These manual checks are urgent because operators need immediate results.

If the monitoring sweep is running, manual checks should not become slow or blocked behind background tasks.

Implement a high-priority execution path for manual checks if needed.

Recommended priority order:

Manual single-site / single-feature checks
Manual single-server checks
Manual full sweep
Automatic monitoring sweep
Non-critical scheduled background tasks

Manual operator actions should feel fast and responsive.

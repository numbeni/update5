The goal is simple:

==================================================
CORE PRODUCT LOGIC

The Network Connectivity system must be based on backend-side ping checks.

The backend should run real ping commands against a small predefined list of reliable targets.

Hardcoded default targets:

google.com
soft98.ir
varzesh3.com

Important:
The actual target values may be stored as:

google.com
soft98.ir
www.varzesh3.com

Do not store the protocol as part of the ping target because ping should target a hostname/domain, not a URL.

If current data includes:

https://soft98.ir/
https://www.varzesh3.com/

normalize them internally to:

soft98.ir
www.varzesh3.com

The UI can display them nicely, but the backend ping command must use only the hostname.

==================================================
IMPORTANT BEHAVIOR CHANGE

Do NOT run connectivity checks constantly every few seconds while the system is healthy.

We do NOT want permanent aggressive background checking.

Instead, connectivity checks should run in these situations:

After each completed monitoring sweep
When several consecutive DOWN results happen during live monitoring
When the operator manually clicks “Run Check” in the Network Connectivity page
When the system is already OFFLINE, then retry every few seconds until it becomes ONLINE again

This means:

When ONLINE, do not check aggressively all the time.
Run one connectivity check after each sweep.
Run an emergency connectivity check if the monitoring engine starts seeing multiple consecutive DOWN results.
When OFFLINE, retry quickly, for example every 5 seconds.
==================================================
PING DECISION RULE

Connectivity decision must be simple:

Run ping checks against the configured targets.

If at least ONE target responds successfully:

global internet status = ONLINE

If ALL targets fail:

global internet status = OFFLINE

Important optimization:
As soon as one target succeeds, stop checking the remaining targets for that run.

Example:

Check google.com:

if ping succeeds:
mark Internet = ONLINE
do not continue pinging soft98.ir and varzesh3.com for this check cycle

If google.com fails:

check soft98.ir

If soft98.ir succeeds:

mark Internet = ONLINE
stop checking remaining targets

If google.com, soft98.ir, and varzesh3.com all fail:

mark Internet = OFFLINE
==================================================
PING IMPLEMENTATION

Use real OS-level ping from the backend.

Linux / Replit / production example:

ping -c 1 -W 3 google.com

Windows example:

ping -n 1 -w 3000 google.com

The code should detect the platform and use the correct ping command.

A target is successful only if the ping command exits successfully.

A target fails if:

ping exits with non-zero code
command times out
host cannot be resolved
target is unreachable

Do not use browser navigator.onLine as the source of truth.

Do not use frontend-only checks as the source of truth.

Do not use TCP / DNS / HTTPS fallback as the main decision logic.

The final global state must be based on ping success/failure.

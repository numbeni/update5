 

## 🧾 FINAL REPLIT PROMPT (UPDATED)

You are a senior backend engineer and system architect.

Build a production-grade **Website Monitoring & Incident Detection Platform** for a hosting company.

This is a **local web-based NOC-style monitoring system** that continuously monitors multiple customer websites and detects DNS issues, HTTP failures, SSL problems, latency issues, and server outages.

The system must be modular, scalable, and ready for future production use.

---

# 🎯 CORE OBJECTIVE

The system must:

* Continuously monitor a dynamic list of websites
* Detect and classify failures (not just UP/DOWN)
* Provide a real-time web dashboard (local hosted)
* Store historical monitoring data
* Generate smart alerts only when real incidents happen
* Be structured for scalability and future SaaS conversion

---

# 📄 WEBSITE INPUT SYSTEM (IMPORTANT)

The system MUST load monitored websites from an external file.

## Option 1 (Simple MVP):

`sites.txt`

Example:

```txt id="sites_txt"
https://example.com
https://google.com
https://myshop.ir
```

## Option 2 (Preferred):

`sites.json`

```json id="sites_json"
[
  {
    "name": "Shop 1",
    "url": "https://example.com"
  },
  {
    "name": "Shop 2",
    "url": "https://google.com"
  }
]
```

The system must dynamically read this file at runtime.

---

# 🧠 MONITORING ENGINE REQUIREMENTS

For each site, perform:

## 1. DNS Monitoring (Gold DNS ready)

* Resolve domain using multiple resolvers (mock or pluggable module)
* Detect:

  * DNS failure
  * propagation issues
  * slow resolution

---

## 2. HTTP Monitoring

* Perform GET request
* Capture:

  * status code
  * response time
  * timeout errors

Detect:

* DOWN (no response)
* SERVER ERROR (500+)
* SLOW (high latency)

---

## 3. TCP Check

* Check ports 80 and 443 availability

---

## 4. SSL Monitoring

* certificate validity
* expiration date
* hostname mismatch

---

# 🧠 INCIDENT DETECTION ENGINE

Convert raw checks into structured incidents:

Example output:

```json id="incident_model"
{
  "domain": "example.com",
  "dns_status": "OK",
  "http_status": 500,
  "response_time": 3.2,
  "ssl_status": "VALID",
  "incident_type": "Application Issue",
  "severity": "CRITICAL",
  "overall_status": "DOWN"
}
```

---

# 🚨 SMART ALERT SYSTEM

Implement intelligent alerting:

Rules:

* Do NOT alert on single failure
* Trigger alert only after 3 consecutive failures
* Deduplicate alerts
* Track alert lifecycle:

States:

* OPEN
* ACKNOWLEDGED
* RESOLVED

Alerts should be grouped:

* If multiple sites affected by same server → single incident

1. Fix “Not Stable” vs “Down” logic

Refine the status logic as follows:

- A website that has NEVER been successfully reached (never had a successful check) should be `Down` by default.
- `Not Stable` should ONLY be used for websites that:
  - were previously reachable (at least one successful check in the past),
  - but are now failing intermittently.

Suggested logic:
- Add a flag like: `hasEverBeenUp`
- If `hasEverBeenUp = false` and check fails → status = `Down`
- If `hasEverBeenUp = true` and:
  - 1–2 consecutive failures → `Not Stable`
  - 3+ consecutive failures → `Down`
- If a failed site becomes successful again → reset to `Stable`

This must persist in database/state (not just in-memory).

3. Completely redesign “Blocked” detection

Remove ANY hardcoded or manual blocked site list (e.g., banks, e-commerce, etc.).

Blocked must ONLY be used when access is explicitly denied by the target system.

Correct definition:
Blocked = target server is actively rejecting access (IP restriction, geo restriction, WAF, firewall, etc.)

Valid signals:
- HTTP 403
- HTTP 451
- Response contains:
  - “access denied”
  - “forbidden”
  - “blocked”
  - “security policy”
  - “geo restriction”
  - “your IP has been blocked”
  - WAF/Cloudflare challenge

NOT Blocked:
- DNS failure
- timeout
- connection refused
- SSL errors
- 5xx errors

Implementation:
- Create a detection function (e.g., `detectBlockedStatus`)
- Store:
  - `isBlocked`
  - `blockedReason`
  - `lastHttpStatus`
- Show reason in UI and logs
- Auto-clear blocked state if site becomes accessible again
- Add log:
  [BLOCKED] example.com - HTTP 403 (possible geo/IP restriction)

Accuracy is critical. Avoid false positives.

لطفاً روی کد فعلی پروژه تغییرات زیر را اعمال کن، بدون اینکه ساختار اصلی پروژه یا قابلیت‌های موجود خراب شود. قبل از اعمال تغییرات، کد را بررسی کن و بهترین محل‌های پیاده‌سازی را پیدا کن.

زبان و UI پروژه را مطابق سبک فعلی نگه دار.

Features to implement:

1. Website input with name + URL
در بخش اضافه کردن سایت، کاربر بتواند هم «نام سایت» و هم «لینک سایت» را به صورت متنی وارد کند.
مثلاً:
Name: Google
URL: google.com

این اطلاعات باید ذخیره شوند و در داشبورد به جای فقط URL، نام سایت هم نمایش داده شود.

2. Auto-add https://
وقتی کاربر URL وارد می‌کند، اگر آدرس با `http://` یا `https://` شروع نشده بود، سیستم به صورت خودکار `https://` را به ابتدای آن اضافه کند.
این کار هم در لحظه ورود اطلاعات و هم قبل از ذخیره‌سازی/تست سایت انجام شود تا URL همیشه استاندارد باشد.

3. Live CMD-style activity log
یک بخش لاگ کلی شبیه CMD/Terminal در داشبورد یا صفحه مخصوص monitoring اضافه کن که تمام اتفاقات مهم سیستم را در لحظه نمایش دهد.
لاگ‌ها باید شامل مواردی مثل این باشند:
- شروع تست سایت
- URL در حال بررسی
- نتیجه HTTP status
- زمان پاسخ‌دهی
- DNS check result
- خطاهای connection / timeout / SSL
- تغییر وضعیت سایت از stable به not stable یا down
- زمان دقیق هر اتفاق

ظاهر لاگ شبیه terminal باشد:
پس‌زمینه تیره، فونت monospace، خطوط جدید به صورت live اضافه شوند.
اگر پروژه WebSocket/SSE دارد از همان استفاده کن، اگر ندارد یک روش سبک و مناسب برای real-time یا near-real-time پیاده‌سازی کن.

4. Increase interval between checks
در حال حاضر تست‌ها خیلی سریع و پشت سر هم انجام می‌شوند.
فاصله زمانی بین check ها را بیشتر کن.
یک مقدار پیش‌فرض منطقی قرار بده، مثلاً هر 60 ثانیه یا هر 2 دقیقه.
همچنین اگر در پروژه تنظیمات وجود دارد، این interval را قابل تنظیم کن.
مطمئن شو loop یا cron فعلی بیش از حد request ارسال نکند.

5. Better dashboard status logic
در داشبورد، اگر یک سایت فقط یک بار down شد، فوراً وضعیت کلی آن را `Down` نشان نده.
به جای آن وضعیت را `Not Stable` نشان بده.
فقط وقتی چند بار پشت سر هم check ناموفق بود، وضعیت `Down` شود.

منطق پیشنهادی:
- اگر آخرین check موفق بود: `Stable` یا `Up`
- اگر 1 یا 2 بار پشت سر هم fail شد: `Not Stable`
- اگر 3 بار یا بیشتر پشت سر هم fail شد: `Down`

اگر threshold مناسبی در پروژه وجود دارد از همان استفاده کن، وگرنه مقدار پیش‌فرض 3 consecutive failures باشد.
این وضعیت باید در دیتابیس/State ذخیره شود تا بعد از refresh هم درست بماند.

6. Custom DNS resolver list
در حال حاضر یک‌سری DNS پیش‌فرض داخل سایت set شده که خوب است و باید حفظ شوند.
اما یک قابلیت اضافه کن که کاربر بتواند DNS resolver های دلخواه خودش را هم اضافه کند.

قابلیت‌ها:
- وارد کردن DNS ها به صورت دستی
- وارد کردن لیست DNS ها به صورت bulk text / textarea
- هر DNS در یک خط یا جدا شده با comma قابل قبول باشد
- validate کن که DNS ها معتبر باشند
- DNS های تکراری حذف شوند
- DNS های پیش‌فرض همچنان باقی بمانند
- DNS های سفارشی در بخش DNS check استفاده شوند
- امکان حذف DNS های سفارشی وجود داشته باشد

- اگر دیتابیس/schema نیاز به تغییر دارد migration مناسب اضافه کن.
- اگر پروژه TypeScript دارد type ها را کامل رعایت کن.
- خطاها را handle کن و اجازه نده UI crash کند.
- برای بخش‌های جدید، validation مناسب اضافه کن.
- وضعیت‌ها و لاگ‌ها باید قابل فهم برای admin/server specialist باشند.
- UI را ساده، مرتب و حرفه‌ای نگه دار.
- بعد از پیاده‌سازی، کد را از نظر خطا بررسی کن و اگر تست یا build command وجود دارد اجرا کن.

Expected output:
- سایت‌ها با name و URL ذخیره شوند.
- URL بدون protocol به صورت خودکار https:// بگیرد.
- live terminal log نمایش داده شود.
- فاصله check ها بیشتر شود.
- dashboard با یک fail سریعاً Down نشان ندهد و ابتدا Not Stable شود.
- DNS resolver های سفارشی قابل افزودن، ذخیره، حذف و استفاده در DNS checks باشند.

Please improve the final recheck phase of the monitoring sweep.

Important:
Do not change the normal manual check behavior.
Do not change the first pass or second pass behavior.
This change must apply ONLY to the final sweep phase where critical/down sites are being confirmed before final incident/status decision.

Goal:
In the final phase of the sweep, for sites that are already considered critical/down and are waiting for final confirmation, run 5 confirmation checks per site instead of only 1.

Reason:
This helps confirm whether the site is truly down or if the previous failures were temporary/flaky network issues.

---

## Required behavior

During the final phase:

For each site in the critical/down final recheck list:

- Run 5 checks
- Collect all results
- Decide final status based on the 5 attempts

Suggested decision logic:

1. If 3 or more out of 5 checks are UP:
   - Treat site as recovered / not truly down
   - Do not mark as final down
   - Do not create critical incident

2. If 3 or more out of 5 checks are DOWN / critical:
   - Confirm site as down
   - Mark as final down
   - Create/update incident if needed

3. If results are mixed:
   - Mark site as unstable / degraded
   - Log clearly that final recheck had mixed results

You may improve this logic if there is a more suitable existing status model.

---

## Scope limitation

This 5x check behavior must apply ONLY to:

- final recheck phase
- down/critical sites
- last phase of automatic sweep

It must NOT apply to:

- manual single-site Run Check
- normal automatic first pass
- second pass
- queue manual checks
- normal site detail diagnostics
